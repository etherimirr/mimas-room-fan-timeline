# 项目说明

## 这是个什么产品

我要做的不是普通博客，也不是普通游戏官网，而是一个基于《未麻的部屋》世界观与氛围的互动叙事网页产品。

这个产品的核心不是“展示内容”，而是让用户以“粉丝”的身份进入一个偶像的个人主页 / 日记站，并逐渐产生被网站记住、被内容回应、被时间线卷入的感觉。

核心体验：
1. 用户不是旁观者，而是以“粉丝”身份进入网站。
2. 网站内容会随着用户首次进入后的时间持续推进，而不是一次性全部展示。
3. 不同用户因为进入时间不同，会拥有不同的解锁节奏和体验路径。
4. 系统会记住用户的行为、阅读进度、分支选择和周目状态。
5. 这个网站会像连载作品一样持续更新，后续可以不断加入新日记、新事件、新分支和新机制。

一句话概括：
这是一个时间驱动、可连载、可分支、可周目的互动叙事网页产品，用户会像进入《未麻的部屋》式的偶像私人网页一样，逐步陷入一个会随时间和行为变化的“活的”网站。

## 世界观 / 剧情目标

这个项目的叙事目标，不是做一个一上来就很可怕的恐怖网页，而是构建一个表面温柔、可爱、日常，实际上逐渐失真的互动叙事空间。

在设定上，用户会以“粉丝”的身份进入一个偶像的个人主页 / 日记站。起初它看起来只是一个普通的个人网站，有近况、日记、留言痕迹和一些生活化内容。但随着时间推进和用户不断返回，网站会逐渐出现越来越强的不协调感，让用户慢慢产生一种心理上的不安：这个网站好像记住了我、在回应我，甚至在观察我。

我希望营造的是一种沉浸式、第一人称的粉丝体验。用户不是站在外部看故事，而是被包含进故事里，逐步经历现实与虚构边界变得模糊的过程。重点不是 jump scare，也不是传统恐怖游戏式的直接惊吓，而是通过时间、重复访问、文本变化、分支反馈和界面细微异常，制造一种“最开始很正常，后来越来越不对劲”的心理失真感。

非常重要的一句：
不要直接做成恐怖游戏 UI；它必须先表现为正常、纯真、日常的偶像主页，再在后续体验中逐步出现异常和错位。

## 风格要求

### 视觉风格
- 整体参考 90 年代早期个人主页 / 日记网页的感觉。
- 要有 old web、简单 HTML、个人站点的质感。
- 颜色以浅粉、白、柔和蓝为主，可以带一点温和的奶油色或浅灰。
- 页面应当显得质朴、真实、略微笨拙，而不是精致商业化。
- 不要做成现代 SaaS dashboard。
- 不要做成高科技、赛博、终端、监控面板风格。
- 不要一上来就使用明显“恐怖游戏 UI”语言。
- 它首先应该像一个真实存在过的偶像个人主页。

### 情绪风格
- 表层情绪应该是温柔、清新、可爱、日常。
- 底层要慢慢渗出不安感、窥视感、身份错位感和被记录的感觉。
- 核心不是直接吓人，而是让人逐渐觉得“越看越不对劲”。
- 用户的心理变化应该从放松、熟悉，逐渐转向怀疑和不安。

### 交互风格
- 交互不能太游戏化，不要像任务系统、关卡系统或 RPG 菜单。
- 用户体验应该尽量接近“我真的在浏览一个人的主页/日记站”。
- 异常效果必须克制，少量但有效。
- 比起大量视觉特效，更重要的是：
  - glitch 文本
  - 某些句子变化
  - 延迟解锁
  - 重复访问后的细节变化
  - 隐藏句子或多出来的一行字
- 异常应该像网页本身出了点问题，或者像内容开始认识用户，而不是像系统弹出一个大招式 jump scare。

### 文案风格
- 前期文案必须自然、日常、生活化，像偶像博客或私人日记。
- 一开始的文字应该让人相信这真的是一个温柔、普通、甚至有点可爱的个人主页。
- 后续再逐渐加入微妙的异常、重复、错位和自我意识。
- 不要从第一屏开始就写得像 horror game。
- 不要过早暴露“这个网站有问题”这件事，要让异常慢慢浮现。

## 系统规则

系统规则：
1. 每个用户必须拥有独立时间线，不能共享统一剧情进度。
2. 所有主要事件都必须基于用户首次进入时间 `first_seen_at` 解锁，时间推进应以“用户进入后经过多久”为核心，而不是只看统一现实时间。
3. 用户状态必须至少记录以下信息：
   - `first_seen_at`
   - `last_seen_at`
   - `flags`
   - `read_entries`
   - `loop`
4. 系统必须能够记住用户行为，并基于这些行为影响后续内容展示与分支结果。
5. 内容必须数据驱动，所有日记、事件、分支、隐藏文本、异常文本都应放在 JSON 或其他数据文件中管理，不能把剧情直接写死在前端组件里。
6. 系统必须支持分支逻辑，例如用户是否留言、是否重复访问、是否在夜间访问，都可以影响内容变化。
7. 系统必须支持周目机制（loop / New Game+），以便后续加入“你之前来过”的元叙事内容。
8. 系统必须支持后续持续新增日记、事件、分支和周目内容，而不需要重写核心逻辑。
9. 前期可以使用 `localStorage` 存储用户状态，但整体结构必须可迁移到数据库或后端存储。
10. 代码结构必须可维护，时间逻辑、状态逻辑、条件判断、事件处理应集中管理，不要把剧情逻辑散落在 UI 组件里。

## 本次先做 MVP

这次先不要做全部功能，而是先完成一个可扩展的 MVP。

MVP 范围：
- 首页
- 日记列表 / 日记详情
- 用户首次进入时间记录
- 基于 `first_seen_at` 的 24 小时后解锁第二篇日记
- 至少一个基于 `flag` 的分支
- 一个克制的异常效果
- 代码结构要为未来继续扩展日记、事件、分支和周目预留空间
